Idea central

El proyecto ya transmite muy bien el universo de marca y varias piezas del MVP, pero todavía conviven dos apps en una: una versión demo/editorial visible y otra base de features más cercana al producto real con Supabase.

Resumen ejecutivo

habitOS ya tiene identidad, dirección y muchas piezas construidas; lo que falta es decidir cuál es la app canónica y cerrar las brechas entre spec, demo y backend real.

Tras revisar la documentación funcional, el brand book, el manual de usuario, la app iOS SwiftUI, el widget y la configuración del proyecto, mi lectura es que habitOS no es una app genérica de hábitos: es una herramienta premium de seguimiento nutricional y coaching, con una capa emocional y estética muy cuidada. El código confirma esa intención, pero también muestra una transición en curso entre una primera app demo muy presentacional y una segunda capa más real conectada a Supabase.

Qué es habitOS

Modelo mental consolidado del producto

Lo que sigue es mi interpretación unificada a partir del spec, README, manual de marca y manual de usuario.

Definición operativa

habitOS es la app de seguimiento diario para clientes de un nutricionista o coach. Su función no es “enseñar nutrición” ni “dar recetas genéricas”, sino ejecutar un plan ya diseñado: dieta, adherencia, chat, diario, progreso, escaneo de productos, listas y seguimiento corporal.

  • Es un producto de retención, no de captación.
  • La promesa es estrategia sostenible, no gamificación agresiva.
  • La experiencia mezcla precisión clínica con una estética editorial y calmada.
  • La IA es motor interno, no protagonista de marketing.

Lo que no es

  • No es una app social.
  • No es una calculadora manual de calorías para el usuario final.
  • No es una simple agenda de hábitos sin contexto nutricional.
  • No es una “fitness app” chillona ni basada en culpa.
Conclusión de negocio

La identidad del proyecto está muy bien pensada. El mayor riesgo no está en la idea, sino en que varias decisiones técnicas y visuales todavía no están completamente alineadas con esa idea.

Hallazgos críticos

Lo más importante que deberíais decidir o corregir

  1. Hay una inconsistencia de dirección visual y de producto entre documentos.
    El spec técnico describe una app oscura por defecto, mientras que el brand book, el README y la app implementada empujan una identidad clara, cálida, editorial y mediterránea. La sensación actual del código coincide mucho más con el brand book que con el spec oscuro.
  2. La navegación principal sigue gobernada por la capa demo.
    El corazón visible de la app todavía depende de HabitOSDataService y modelos root, no de la capa de repositorios y features conectada a Supabase.
  3. Hay duplicación de dominio y riesgo de divergencia.
    Conviven tipos y pantallas equivalentes en Views/, ViewModels/, Services/ y en Features/. Esto aumenta el coste mental y el riesgo de mantener dos versiones distintas del producto.
  4. El flujo de magic link no parece estar completamente cerrado para iOS nativo.
    La documentación oficial de Supabase Swift recomienda manejar deep links con supabase.auth.handle(url) usando onOpenURL o lifecycle UIKit/Scene. En el entry point revisado no aparece esa pieza.
  5. Coach Memory consulta una tabla distinta a la definida en el spec.
    El spec habla de coach_facts; el código consulta coach_memories. Eso sugiere un backend alternativo, una migración sin terminar o una nomenclatura que debo confirmar.
  6. El scanner ya usa Open Food Facts real, pero la lógica de decisión todavía no usa el perfil del usuario.
    El spec promete compatibilidad con alergias, Nutri-Score, NOVA y azúcar. La implementación actual se queda en mostrar el producto y algunos macros.
  7. Las fotos de progreso no siguen todavía el modelo de seguridad final.
    El spec pide bucket privado con URLs firmadas en Supabase; la implementación actual guarda JPGs en documentos locales del dispositivo.
  8. Hay una anomalía llamativa en tests.
    El archivo de unit tests visible parece contener contenido del proyecto Xcode en lugar de XCTest real. Los UI tests sí existen pero son casi plantillas. Esto me hace pensar que la cobertura real hoy es prácticamente nula.

Dependencias externas

Contraste rápido con documentación oficial actual

Supabase Swift

El proyecto usa SPM con supabase-swift. La documentación actual confirma el patrón base del cliente, filtros, CRUD, Storage y Realtime que ya usáis.

  • SupabaseClient(supabaseURL:supabaseKey:) encaja con vuestro SupabaseManager.
  • signInWithOTP es correcto para magic link.
  • Para iOS nativo falta confirmar el manejo de deep links con auth.handle(url).
  • La documentación actual también recomienda escuchar auth state changes en lugar de consultar sesión de forma repetitiva.

Open Food Facts

La API actual sigue en versión 2. La doc oficial insiste en usar un User-Agent identificable y respetar límites de lectura.

  • El código ya envía un User-Agent, lo cual es correcto.
  • La API v2 está confirmada como actual.
  • La app debería tratar sus datos como abiertos pero no garantizados; esto importa para UX y messaging.
  • La lógica de producto debería asumir que no todos los barcodes tendrán datos completos.

HealthKit

La documentación de Apple confirma que HKHealthStore es el punto central para permisos, lectura, escritura y queries.

  • La app ya tiene descripciones de permisos configuradas en el proyecto.
  • El uso de requestAuthorization(toShare:read:) es correcto.
  • La implementación actual comprueba sobre todo pasos, sueño y peso.
  • Si queréis modo realmente productivo, falta completar escritura y sincronización de varios tipos prometidos por el spec.

Respuestas consolidadas

Decisiones de dirección ya aclaradas y criterio recomendado para seguir

Aquí mezclo dos cosas: lo que ya habéis respondido explícitamente y mi criterio técnico cuando la respuesta todavía no es definitiva.
  1. Dirección visual canónica. Queda resuelto: debe mandar el brand book. Por tanto, la referencia visual válida es una app clara, cálida, mediterránea y editorial. El spec técnico debería corregirse para alinearse con esa identidad y dejar de empujar dark mode por defecto como verdad de producto.
  2. Capa root actual. No hay confirmación cerrada de que ContentView + DashboardViewModel + HabitOSDataService sea la shell final, y por el estado del repositorio mi lectura es que debe tratarse como capa transitoria. Recomendación: asumir desde ya que la root actual es un scaffold útil para demo, pero no la arquitectura objetivo.
  3. Memoria del coach en backend. No hay verdad confirmada todavía y es razonable asumir que esa parte puede ni existir aún en base de datos. Decisión práctica: no fijar el nombre de tabla en el dominio hasta inspeccionar la BBDD real. Recomendación: encapsular esta feature detrás de repositorio/protocolo y posponer cualquier acoplamiento duro a coach_memories o coach_facts hasta validarlo.
  4. Magic link en dispositivo real. No está confirmado como flujo cerrado. El hecho de que se use sobre todo un botón de prueba y que no aparezca el cierre completo de deep link en el entry point refuerza que esta autenticación móvil no debe darse por terminada. Recomendación: tratarla como parcialmente implementada hasta probarla end to end en iPhone real.
  5. Rol del modo demo. La intención actual parece ser operativa, no estratégica: está ahí porque faltan conexiones reales con la BBDD y sirve para visualizar la app mientras se construye. Recomendación: mantenerlo como herramienta de desarrollo, QA y demo interna, pero no como pilar de producto definitivo salvo decisión posterior explícita.
  6. Modelos definitivos. Aquí sí conviene tomar una postura fuerte: no deberíais tolerar mucho tiempo más la convivencia entre UserProfile y AppUser, MealPlan legacy y NutritionPlan, o ChatMessage demo y CoachMessage. Mi criterio es converger pronto, antes de ampliar más funcionalidades visibles.
  7. Fotos de progreso. Dado que el producto sigue en construcción, no movería esta feature de inmediato a Supabase Storage si eso obliga a montar backend y seguridad deprisa y mal. Mi recomendación es preparar ya una abstracción de almacenamiento y mantener local-only de forma temporal, con migración posterior a storage privado cuando la base de datos y la auth estén sólidas.
  8. Prioridad inmediata. La respuesta operativa más sensata es: construir el MVP, pero sobre una base corregida. No recomiendo una reescritura total previa, pero sí una refactorización dirigida a unificar shell, modelos y repositorios antes de seguir añadiendo capas duplicadas. En otras palabras: refactorizar lo suficiente ahora para que el MVP que terminéis sea la base buena, no otra rama temporal.
Mi recomendación

Con tus respuestas, la dirección correcta ya se ve bastante clara: conservar el lenguaje de marca actual, tratar la shell demo como transitoria, usar el modo demo sólo como soporte de construcción y converger pronto hacia una sola capa de modelos y repositorios. El siguiente paso sano no es “hacer una app nueva”, sino convertir la app visible en una versión cada vez menos mockeada y menos duplicada.

Ledger obligatorio

Handoff directo

He creado NEXT_COPILOT_HANDOFF.md como documento de arranque directo para el siguiente Copilot. Su función es evitar que cada nuevo agente tenga que reconstruir desde cero qué debe leer, cómo debe trabajar y qué documentos debe mantener vivos.

Regla operativa básica

Ningún futuro agente debería necesitar que le repitas qué es HabitOS, qué estética manda o cómo debe registrar su trabajo. Si cambia algo en el repo, debe reflejarlo en el ledger; si termina algo, debe dejar hora, evidencia y validación.

Comportamiento esperado del otro Copilot

  1. Leer primero el ledger y el intelligence report.
  2. Leer NEXT_COPILOT_HANDOFF.md como briefing inicial de takeover.
  3. Añadir o actualizar la tarea correspondiente antes de editar archivos.
  4. Marcar la tarea como IN PROGRESS al empezar.
  5. Marcarla como DONE al terminar con timestamp real.
  6. Documentar archivos tocados, validaciones realizadas y follow-ups descubiertos.
  7. Actualizar los documentos de contexto si el entendimiento del proyecto o la guía de takeover cambia.

Resumen narrativo para IA 
habitOS es una app iOS de seguimiento nutricional premium para clientes de coach/nutricionista.

Objetivo del producto:
- ejecutar un plan nutricional ya diseñado
- registrar adherencia, agua, peso, diario y progreso
- hablar con un coach en tiempo real
- escanear productos y decidir compatibilidad
- construir una experiencia premium, calmada, editorial y no fitness-agresiva

Estado del repositorio:
- la identidad y el posicionamiento están bien definidos
- existe una capa visible basada en datos demo
- existe una capa de features más real con Supabase, pero no gobierna toda la app
- varias features están construidas pero integradas parcialmente

Decisiones ya aclaradas:
- la dirección visual canónica debe seguir el brand book
- la shell demo actual debe leerse como transición, no como destino ideal
- el modo demo existe para suplir la falta de backend plenamente conectado
- conviene converger pronto a un único conjunto de modelos
- la refactorización debe apoyar al MVP, no frenarlo indefinidamente

Riesgos principales:
- duplicación de modelos y pantallas
- inconsistencia entre spec oscuro y marca/implementación clara
- dudas sobre flujo de auth profundo y tabla real de memoria del coach
- tests muy flojos o potencialmente corruptos

Hipótesis fuerte:
- el proyecto está en transición de dashboard demostrable hacia producto real conectado a backend
